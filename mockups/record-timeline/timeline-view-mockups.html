Timeline view - static mockups for review

A first-class TIMELINE view type for the record index page: one row per record, a bar spanning its start→end date fields, on a horizontally scrolling time axis. Architecture follows the CALENDAR view precedent (view entity fields + CHECK constraint, record-index dispatch, view picker, SDK manifest). UX semantics come from the gantt component twenty-eng validated in production inside a sandboxed front component.

Three variants below, rendered inside the real record index chrome. All colors, spacing, radii and type are taken verbatim from twenty-ui light-theme tokens (Inter is embedded from the repo's own @fontsource/inter). Dark theme uses the same token slots and is not re-mocked.

Variant 1

Flat rows with zoom presets

The direct translation of twenty-eng's gantt: every record is one row, ordered by the view's sorts. A quarter navigator and Month / Quarter / Year zoom presets sit in a slim toolbar exactly where the calendar view puts its month navigator. The left column is a sticky record chip (opens the side panel) with a related-count pill.

  • Bar color encodes record state (on-track / at-risk / off-track); dashed hollow bar = scheduled but not started; green diamond on the end date = completed; purple diamond = single-date milestone record.
  • Bars extending past the window are clipped square with an overflow arrow, like GitHub roadmap.
  • Hover shows a highlight ring and a tooltip with exact dates and duration.

Variant 2

Grouped lanes (record groups)

Same rows, but grouped into collapsible lanes using the record-group mechanism Kanban already uses (group-by any SELECT field - here stage). Group headers reuse the exact Tag treatment from board column headers. A collapsed group keeps a thin summary span showing the min→max range of its hidden bars.

  • Groups collapse/expand independently; state persists per view like Kanban column state.
  • Bar colors still encode record state, group tag encodes the grouping field - no double encoding on the bars themselves.
Variant 3

Density + unscheduled bucket + empty state

How the view behaves when data is imperfect: compact 28px rows for larger sets, view-bar filters and sorts applying exactly as in table/kanban (chips row shown), and records missing either date collected in an Unscheduled section at the bottom. Hovering an unscheduled row offers a ghost "Set dates" affordance at the today line.

Variant 3b

Empty state

Consistent with other views' placeholder pattern: explains the two ways out (add dates, or clear filters).

Recommendation

Ship v1 as Variant 1 + the unscheduled bucket and density treatment of Variant 3. They compose naturally (the bucket is just a partition of the same flat list) and cover everything the twenty-eng Companion needs to delete its custom gantt. Zoom presets ship in v1 since the axis renderer needs windowing math anyway.

Grouped lanes (Variant 2) land as a fast-follow, reusing Kanban's record-group state so a view can toggle grouping without a new view. This mirrors how CALENDAR shipped scoped-down first. Bar-drag date editing and color-by-select (maps to Companion's confidence field) stay in the Phase 4 stretch bucket as agreed.

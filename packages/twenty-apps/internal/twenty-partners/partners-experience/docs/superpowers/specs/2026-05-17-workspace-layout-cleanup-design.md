# Twenty Partners — Workspace Layout & Workflow Cleanup

**Date**: 2026-05-17
**Status**: Design — ready for implementation plan
**Author**: Rashad Karanouh
**Context**: The current twenty-partners app has 2 nav items (Partners, Partner deals) buried under all native Twenty objects in the sidebar. Working in the partner CRM means scrolling past Tasks, Notes, and Workflows that the partner workflow never touches. We need a focused workspace — partner-related items at the top, a way to restrict access for partner-facing users, and dedicated views for the two operational moments that matter: "deal arrived, waiting for a partner" and "deal matched, here's what happened next." All configured as code so we never rebuild this by hand.

---

## What this design changes

| Area | Before | After |
|---|---|---|
| Sidebar nav (admin) | Partners + Partner deals at positions 10–11, below all natives | Waiting for match → All partner deals → Matches overview → Partners at positions 0–3, above natives |
| Restricted access | None — only Admin role exists | Two new roles: `Twenty Partner Ops` (internal teammate, full data access on partner objects) + `Partner` (placeholder for future external self-service, currently un-assignable until RLP ships) |
| Partners list view | Columns: name, status, availability, languagesSpoken, lastMatchAt | + servedGeos, + deploymentExpertise; filter hides REJECTED |
| Partner record page | Auto-generated by Twenty | Explicit `RECORD_PAGE` page layout (Fields tab; relation tabs surface via Twenty's default behavior) |
| Opportunity workflow views | One ("Partner deals") showing all eligible deals | Three: waiting queue, full list (kept as catch-all), matches kanban |
| Demo data | Only 6 seeded partners; no opportunities | New seed script creating 3 companies, 3 persons, and 15 opportunities (40% partner-eligible across kanban columns + 60% raw pipeline you can flip to test the live matching trigger) |

Admin role keeps full visibility — nothing hidden from the workspace owner.

---

## Architecture

Everything is config-as-code via `twenty-sdk/define`. No new runtime logic. No new HTTP endpoints. The cleanup is delivered as ~6 new files and ~3 edits to existing files, all under `src/`.

### Constraints surfaced upfront

**Navigation is additive only.** `defineNavigationMenuItem` adds sidebar items but cannot hide native ones. Hiding native objects for non-Admin users is achieved via **role permissions** (`defineRole` with `objectPermissions`) — Twenty hides nav items for any object the user has no permission on.

**No row-level permissions in the SDK today.** `ObjectPermissionManifest` has only object-wide booleans (`canReadObjectRecords`, etc.) — there is no `recordFilter` field. This blocks the "Partner sees only own records" goal. We work around this by defining the Partner role now as a structural placeholder with `canBeAssignedToUsers: false` — when Twenty ships record-level filters, the role is already wired and just needs the filter added + the boolean flipped. The original Phase 5 deferral (RLP spike) remains valid as a follow-up.

**No member-seeding as code.** Twenty SDK does not expose workspace member invitation/assignment via post-install or any other hook. Creating test users for the non-Admin roles is a documented manual flow (see the testing guide).

## Components

### 1. Roles (Admin + 2 NEW)

Three roles total. Only the latter two are new code.

#### 1.a Admin — **NO CHANGES**
Twenty's built-in Admin role. The workspace owner (`tim@apple.dev` in local dev) is already assigned. Has full access to everything — partner objects, native objects, settings, workflows. This is what you (Rashad) use when developing and operating the workspace. We do not modify it.

#### 1.b Twenty Partner Ops — **NEW**
**File**: `src/roles/twenty-partner-ops.role.ts`

The internal Twenty teammate role. Imagine a partner-success manager at Twenty whose job is matching deals and onboarding partners. They need full data access to the partner workflow but should not be tinkering with workspace settings or unrelated objects.

- `defineRole({ label: 'Twenty Partner Ops', canUpdateAllSettings: false, ... })`
- `objectPermissions`:
  - `Partner` — read + update + softDelete
  - `Company` — read + update
  - `Person` — read + update
  - `Opportunity` — read + update
- No entry for Tasks/Notes/Workflows/Messages/etc. → no access to those objects
- `canBeAssignedToUsers: true`

**Side effect**: a Partner Ops user opening the sidebar sees only the 4 objects listed above + the 4 partner views at the top. Tasks/Notes/Workflows nav items don't render because the user has no permission on those objects.

#### 1.c Partner — **NEW (placeholder)**
**File**: `src/roles/partner.role.ts`

A placeholder for the eventual external-partner self-service role. The intent is "a contact at Nine Dots Ventures logs into Twenty (or a portal that proxies to Twenty) and sees only their own Partner record + the Opportunities matched to them." **Twenty does not yet support row-level filtering**, so this role would currently leak all partner data if assigned. We define it now so the structural wiring (role exists, has a stable UUID, nav items can reference it later) is in place for when RLP ships.

- `defineRole({ label: 'Partner', icon: 'IconBuildingStore', ... })`
- `description`: `'PLACEHOLDER. External partner self-service role. Sees ALL Partner/Opportunity records today because Twenty does not yet support row-level record filtering. When RLP ships, scope these permissions to records owned by the assigned user. DO NOT assign to real external partners until then.'`
- `objectPermissions`:
  - `Partner` — read + update (so when scoped, the partner can edit their own profile)
  - `Opportunity` — read (so when scoped, they see only matched deals)
  - `Person` — read (so they can see other contacts at their firm)
  - `Company` — read
- `canBeAssignedToUsers: false` (initial safety — flip to true when RLP lands)
- `canUpdateAllSettings: false`

**Why `canBeAssignedToUsers: false`**: prevents accidental assignment in the UI today. The role exists, the permissions are wired, the UUID is stable — but Twenty's member-assignment dropdown won't show it as an option. When you're ready to flip it on, change that one boolean.

### 3. New views (2 NEW + 2 EDITS)

#### `src/views/waiting-for-match.view.ts` (NEW)
- **Type**: TABLE
- **Object**: Opportunity
- **Filters**:
  - `partnerEligible IS true`
  - `partner IS_EMPTY` (using `ViewFilterOperand.IS_EMPTY` on the partner relation field)
- **Sort**: `createdAt ASC` (oldest waiting deals at top — they need attention first)
- **Columns**: `name`, `company` (relation), `createdAt`, `designDocStatus`, `partnerEligible` (visual confirmation)

#### `src/views/matches-overview.view.ts` (NEW)
- **Type**: KANBAN
- **Object**: Opportunity
- **Group by**: `matchStatus` field (via `mainGroupByFieldMetadataUniversalIdentifier`)
- **Filter**: `partner IS_NOT_EMPTY` (any opportunity that has a partner assigned)
- **Columns** (the kanban swimlanes are the matchStatus options: DELIVERED, INTRO_SENT, ENGAGED, IMPLEMENTING, COMPLETED, STALLED, CANCELLED — auto from field options)
- **Card fields**: `name`, `partner`, `introSentAt`

#### `src/views/all-partners.view.ts` (EDIT)
- Add columns: `servedGeos` (position 5), `deploymentExpertise` (position 6)
- Add filter: `status IS_NOT 'REJECTED'` (hide rejected partners from the default view)
- Keep existing columns and sort

#### `src/views/partner-eligible-opportunities.view.ts` (EDIT)
- Rename `name`: `'Partner deals'` → `'All partner deals'`
- No other changes (filter and columns stay the same)

### 4. Partner record page layout (NEW)

**File**: `src/page-layouts/partner-record-page.page-layout.ts`

#### What is the "record page"?
When you click on a single partner in the Partners list (e.g. "Nine Dots Ventures"), Twenty opens a detail page for that one record. That page has a header, a fields panel on the left, and tabs on the right for related records (Opportunities assigned to this partner, Persons working at this partner) and activity. Today this page is auto-generated by Twenty using defaults. A **page layout** is the as-code way to declare exactly what tabs and widgets appear on that detail page.

#### Visual mockup of the target layout

```

│  [back]   Nine Dots Ventures   [● ACTIVE] [● AVAILABLE]   [edit ⋮]   │
├──────────────────────────────────────────────────────────────────────┤
│                                                                       │
│  ┌────────────────────┐  ┌────────────────────────────────────────┐  │
│  │ Profile fields     │  │ [Profile] [Opportunities] [Persons]    │  │
│  │                    │  │ ─────────                              │  │
│  │ name      Nine ... │  │                                        │  │
│  │ slug      nine-... │  │   (active tab content renders here)    │  │
│  │ status    ACTIVE   │  │                                        │  │
│  │ avail.    AVAIL.   │  │   Profile tab:  same fields as left    │  │
│  │ langs     EN, FR   │  │   panel, full-width for editing.       │  │
│  │ geos      EU, MENA │  │                                        │  │
```

#### Tabs defined explicitly in the page layout

1. **Profile** (position 0, icon `IconList`, layoutMode `VERTICAL_LIST`)
   - Single widget of type `'FIELDS'` with `configurationType: 'FIELDS'`
   - This is the same widget Twenty's default record page uses — by defining it ourselves, we own the tab order

#### Relation tabs (Opportunities, Persons)

Twenty surfaces relation tabs **automatically** alongside any custom page layout. The `opportunities` and `persons` reverse-relation fields on Partner generate tabs for free. We do not need to add `RECORD_TABLE` widgets for them.

**Verification step during implementation**: confirm that defining a `RECORD_PAGE` page layout does NOT suppress the auto-generated relation tabs. If it does (i.e. Twenty renders ONLY the tabs we declare), we add explicit `RECORD_TABLE` widgets for Opportunities and Persons. Otherwise we leave the layout minimal.

#### Why this is just a one-tab layout for now

Compare to `carte-postale/page-layouts/quote-record-page.page-layout.ts` — they define Fields + a custom Summary tab using a `FRONT_COMPONENT`. We could add similar later (e.g. a "Match stats" summary card showing total deals matched / completion rate / pipeline value). For this spec we keep it foundational: the layout exists in code, future widgets are a one-file edit away.

### 6. Demo data seed script (NEW)

**File**: `src/scripts/seed-pipeline-demo.ts`
**Run**: `yarn vitest run --config vitest.seed.config.ts` (same command as the partners seed — vitest discovers both files in `src/scripts/`)

The 6 partners we already seed give us a Partners view to look at. But "Waiting for match", "All partner deals", "Matches overview", and the Partner record page are all empty until we have Opportunities. To verify every surface renders well, we need realistic data spread across every matchStatus column.

This script seeds three layers of data, all idempotent (by slug/name):

**Layer 1: Companies (3)**
Realistic prospect companies that would plausibly need a CRM implementer:
- `Acme Real Estate` (FR market — pairs naturally with Nine Dots Ventures)
- `Helix Bio` (US/EN market — pairs naturally with Elevate Consulting)
- `Sunrise Logistics` (APAC market — pairs naturally with Meridian Craft)

**Layer 2: Persons (3)** — one primary contact per company, with the `companyId` relation set.

**Layer 3: Opportunities (15 total — 40% partner-eligible, 60% raw)**

We deliberately leave the majority of seeded opportunities with `partnerEligible: false` so you have plenty of raw pipeline records to play with — flipping their toggle is the cleanest way to test the live matching trigger end-to-end.

**Partner-eligible (6 opportunities = 40%)** — populate the partner views with realistic data:

| Count | matchStatus | partnerEligible | partner | Where it shows up |
|---|---|---|---|---|
| 1 | (null) | `true` | (null) | "Waiting for match" view |
| 1 | `DELIVERED` | `true` | seeded partner | Matches overview > DELIVERED |
| 1 | `INTRO_SENT` | `true` | seeded partner (+ `introSentAt` set) | Matches > INTRO_SENT |
| 1 | `ENGAGED` | `true` | seeded partner | Matches > ENGAGED |
| 1 | `IMPLEMENTING` | `true` | seeded partner | Matches > IMPLEMENTING |
| 1 | `COMPLETED` | `true` | seeded partner | Matches > COMPLETED |

**Raw / playable (9 opportunities = 60%)** — `partnerEligible: false`, no partner, no matchStatus. These look like normal pipeline opportunities. They do NOT appear in any partner view. You flip `partnerEligible` to true on whichever one(s) you want to test the live matching trigger on.

**Summary by surface**:
- Waiting for match: 1 seeded (plus any raw opp you toggle eligible)
- All partner deals: 6 seeded
- Matches overview: 5 seeded across 5 kanban columns (STALLED + CANCELLED columns start empty — drag cards there manually to test)
- Standard Opportunities view (native, visible to Admin): all 15 visible

**Important: bypassing the matching trigger**
The matching logic function fires on `opportunity.updated` when `partnerEligible` flips false→true. Creating an opportunity with `partnerEligible: true` at creation time is an `opportunity.created` event — the trigger does NOT fire. This is what we want for seeding the matched opportunities: we set `partnerId` + `matchStatus` directly in the create mutation. The 1 "waiting" seeded opportunity is also created with `partnerEligible: true, partner: null` — it sits in the queue but the trigger ignores it (no update event). To test the live trigger: pick any of the 9 raw opportunities, flip `partnerEligible` to true → matcher fires within ~5s → opportunity gets a partner assigned and `matchStatus: DELIVERED`.

**Pattern reuse**: mirrors `src/scripts/seed-marketplace-partners.ts` exactly:
- One `describe` block with one `it` block
- `CoreApiClient` instantiated zero-arg
- Idempotent slug/name check before each create
- Logs `[seed] created ...` or `[seed] skip ...`

**Distribution strategy across seeded partners**: round-robin across the 6 partners (Nine Dots, Elevate, W3Villa, Act Education, NetZero, Meridian). This produces ≥1 matched opportunity per partner so every Partner record's Opportunities tab has content.

## How to test the roles

This is a critical section because two of the three roles can only be tested by logging in as a different user. The local Twenty dev workspace ships with **one default user** (`tim@apple.dev` / `tim@apple.dev`, assigned the Admin role). Testing other roles means creating additional workspace members.

### Test 1: Admin role (default user)
No setup required. Log in as `tim@apple.dev` / `tim@apple.dev`. You should see everything — both native objects and partner objects, plus full Settings access.

### Test 2: Twenty Partner Ops role
**Setup (one-time, manual UI step — Twenty SDK cannot seed workspace members yet)**:
1. Logged in as Admin (`tim@apple.dev`), go to **Settings → Members**
2. Click **Invite member**. Enter email `partner-ops-test@apple.dev` and a name
3. The local Twenty instance generates an invite link. **Copy the invite link** (do NOT rely on email delivery — local dev does not send real emails)
4. Open the invite link in a **private/incognito window**, complete signup with any password (e.g. `partner-ops-test@apple.dev` / `partner-ops-test@apple.dev` for consistency)
5. Back as Admin in your main window: **Settings → Members** → find `partner-ops-test@apple.dev` → assign role **Twenty Partner Ops**. **Remove** the default Admin role from this user (Twenty assigns Admin by default to new members)
6. In the incognito window, refresh the workspace

**Expected**:
- Sidebar shows only: Waiting for match, All partner deals, Matches overview, Partners, Companies, People, Opportunities
- Tasks, Notes, Workflows nav items are NOT visible
- Settings menu is limited (no role/billing/data-model access — `canUpdateAllSettings: false`)
- Clicking any partner record works; full read/edit of the 4 permitted objects works

### Test 3: Partner role (placeholder)
The Partner role has `canBeAssignedToUsers: false`, so it **cannot be assigned via the UI today**. This is intentional safety to prevent accidentally leaking all partner data to an external user before row-level scoping ships.

**To smoke-test the role definition itself** (does it sync, are permissions wired correctly?):
1. After `yarn twenty dev --once`, query the role exists: open `http://localhost:2020/settings/roles`, confirm a "Partner" role is listed with the placeholder description visible
2. The "Assign to member" button should be disabled or absent for this role
3. The role's permissions screen should show: read+update on Partner, read on Opportunity/Company/Person

**When Twenty ships RLP** and you're ready to flip the role live:
1. Edit `src/roles/partner.role.ts` → set `canBeAssignedToUsers: true`
2. Add the row-level filter (exact API TBD by Twenty)
3. Follow the same invite flow as Test 2 to create a test partner user and assign the role
4. Verify the user sees only their own Partner record and only Opportunities where `partner = their partner record`

### Cleanup between test sessions
- The invited `partner-ops-test@apple.dev` member persists in the workspace until manually removed
- To reset: **Settings → Members** → remove the test member
- Or accept that test members live in the workspace permanently (low cost — local dev only)

### What we do NOT have (and why)
- **No script-as-code member seeding.** Twenty's SDK doesn't expose member invitations through `definePostInstallLogicFunction` or similar. The invite-and-assign step is unavoidably manual in the UI today. This means test setup for the non-Admin roles is documented here rather than automated.

## What this design does NOT do

- **Does not modify the Admin role.** Admin keeps full sidebar visibility — this is intentional. You (Rashad) continue to see everything as `tim@apple.dev`.
- **Does not assign the Twenty Partner Ops role to anyone.** That's a one-time manual step (invite → assign) per the testing guide above.
- **Does not enable the Partner role for assignment.** `canBeAssignedToUsers: false` gates it until Twenty ships row-level permissions. The role is wired now so flipping it live is a one-line change later.
- **Does not implement row-level scoping for the Partner role.** Twenty SDK does not support `recordFilter` on `ObjectPermissionManifest` today. The Partner role exists as a placeholder; actually scoping it to "own records only" is a separate spec (Phase 5 — RLP spike) and depends on Twenty platform work.
- **Does not add a custom summary card to the Partner record page.** The Fields-only page layout is the foundation; widgets can be added later in one file edit.
- **Does not delete the existing "Partner deals" view.** Kept as a catch-all per user preference; just renamed to "All partner deals" for clarity.
- **Does not seed test workspace members programmatically.** Twenty SDK doesn't expose member invitations. The invite flow for test users is manual and documented in the testing guide.

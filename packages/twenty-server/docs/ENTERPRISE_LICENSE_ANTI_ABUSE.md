# Enterprise License Validation & Anti-Abuse Telemetry — Design Note

> **Status:** Proposal / not yet implemented. This note captures agreed design
> decisions so the payload, token changes, and detection rules are on record
> before code lands.

## 1. Goal & threat model

Twenty Enterprise features (SSO, row-level security, audit logs) are gated by a
Twenty-signed license. For **self-hosted** deployments the operator controls the
runtime, so the gate can never be made tamper-proof — anyone determined can patch
the check and rebuild. We therefore split the problem into three layers and are
honest about what each can achieve:

1. **Prevention** — the only valid license is one Twenty signed (existing signed
   validity token; hardened in §4). Stops forgery, not patching.
2. **Containment** — short-TTL, instance-bound tokens so an issued license cannot
   be copied between instances or used long after revocation (§4).
3. **Detection & attribution** — *this note*. Surface signs of abuse from the data
   licensed instances already send to keep working (§3, §5).

**What detection can and cannot do.** Telemetry is generated by code on the
operator's machine; a competent cracker disables or spoofs it. So this layer
catches the **careless**, supports **attribution/outreach**, and powers
**license administration** — it is explicitly *not* a runtime prevention and must
not be presented as one. Detection value comes from **correlating the honest
majority's reports** (which are truthful) — a cheater can hide *himself* but
cannot fake the population-level pattern of sharing/overage.

## 2. Channel scope (important)

This applies **only** to the enterprise license channel, which fires **only when
an `ENTERPRISE_KEY` is present**. Confirmed in code: `refreshValidityToken()` and
`reportSeats()` early-return with no network call when no key is configured
(`enterprise-plan.service.ts`), and the cron
(`enterprise-key-validation.cron.job.ts`) sends nothing without a key.

Consequences:

- **Free / community (AGPL) instances never transmit on this channel.** No
  community telemetry is involved.
- The channel is governed by the **commercial enterprise agreement**; the terms
  must disclose what is transmitted (link this note).
- **Do not** place any of these signals into the general telemetry path
  (`TELEMETRY_ENABLED`, default `true`, community-facing). Keep the two channels
  strictly separate.

## 3. Reported payload

Sent on the existing enterprise calls to `twenty.com` (`/validate`, `/seats`).
Endpoints live in `packages/twenty-website/src/app/api/enterprise/*/route.ts`.

| Field | Type | Source (twenty-server) | Purpose | Sensitivity |
|---|---|---|---|---|
| `subscriptionId` | string | parsed from `ENTERPRISE_KEY` | primary correlation key | low |
| `serverId` | string (uuid) | `SERVER_ID` | instance identity; sharing & clone detection | low |
| `serverUrl` | string | `SERVER_URL` | instance endpoint; "one serverId on many URLs" | medium (identifying) |
| `workspaceCount` | int | count of workspaces | plan consistency | low |
| `activeUserWorkspaceCount` | int | `getActiveUserWorkspaceCount()` | seats + reconciliation (see note) | low |
| `distinctUserCount` | int | count of distinct `User` rows | sharing/overage delta vs memberships | low |
| `enterpriseFeatures` | `{ sso, rls, auditLogs: bool }` | active gated features | plan consistency | low |
| `appVersion` | string | `APP_VERSION` | fleet/support; coarse anomaly | low |
| `nodeEnv` | string | `NODE_ENV` | dev-mode-in-prod flag | low |
| `telemetryEnabled` | bool | `TELEMETRY_ENABLED` | consistency | low |
| `adminContactEmail` | string (**single**) | primary workspace owner/admin | **license administration & support contact only** | PII |
| `imageDigest` | string (optional) | running image digest if available | low-confidence modified-build lead | low |
| `sentAt` | timestamp | clock | cadence (mostly derived server-side) | low |

### Notes on specific fields

- **`activeUserWorkspaceCount` == seats.** The enterprise seat count already
  reported is exactly `userWorkspaceRepository.count({ deletedAt: IsNull() })`
  (`enterprise-key-validation.cron.job.ts:getActiveUserWorkspaceCount`). It is the
  same number as "memberships" — so we send it **once** under this name and keep
  using it to drive the Stripe subscription quantity (website `/seats` route). Do
  not also send a separate `seatsReported`.
- **The three counts that carry independent signal** are `workspaceCount`,
  `activeUserWorkspaceCount` (memberships), and `distinctUserCount`. Their deltas
  are the interesting part (a user may belong to multiple workspaces, so distinct
  users ≤ memberships).
- **`adminContactEmail` is PII and is a contact field, not a fraud signal.** It is
  trivially spoofable, so it does **not** contribute to abuse detection. We keep it
  to **one** address for legitimate account administration / outreach. The
  authoritative licensee contact remains the **billing email captured at
  checkout**, keyed by `subscriptionId`; this field only records the operating
  admin where it differs. Treat under the DPA: purpose-limited, retention-limited,
  secured.
- **`imageDigest`** is a self-reported value (white-box: spoofable). Keep it as a
  low-confidence lead only; authenticity is established by **external**
  verification of the published signed image (cosign/Sigstore), never by trusting
  this field.

### Explicitly NOT collected

Record/object/field data; any member emails beyond the single admin contact; CRM
content of any kind; IP/geo beyond what an HTTP request incidentally exposes;
persistent device fingerprints; client-computed "self-integrity" hashes (false
comfort — white-box defeatable).

## 4. Validity-token hardening (containment)

**Today:** `EnterpriseValidityPayload = { sub, status, iat, exp }`, ~30-day TTL,
signed by Twenty's key (`twenty-website` `signValidityToken`), verified in
`enterprise-plan.service.ts` `verifyJwt`. On refresh failure the existing token
keeps working until `exp`.

**Proposed:**

1. **Instance binding.** Add `serverId` (and optionally `licensedDomain`) to the
   signed payload. The server verifies `payload.serverId === SERVER_ID` and rejects
   on mismatch. A token then cannot be copied to another instance.
2. **Shorter TTL** (e.g. 7 days) so revocation/expiry actually bites. The refresh
   cron runs well within that, so legitimate instances are unaffected.
3. **Air-gapped customers** get explicitly longer-lived tokens **by agreement**
   (opt-in) — the *default* stays short.

**Why this matters for detection:** short TTL forces every deployed copy to keep
calling `/validate` to stay functional. Combined with `serverId`, that turns the
license channel into a signal the cracker *needs* to keep emitting — which is what
makes the cadence/concurrency rules in §5 work. Binding also means a leaked token
is useless elsewhere and revocable.

> Cross-repo: the **signing** side (`signValidityToken`, and the `/validate`
> handler that must receive `serverId` to bind it) lives in `twenty-website`. The
> verifying/reporting side lives in `twenty-server`.

## 5. Server-side detection rules

Run on Twenty's side over the `/validate` + `/seats` stream. Output is a
**risk score / flags per subscription for manual review and outreach** —
conversion first, enforcement second. **No automated lockout** (false positives
must not break legitimate customers).

- **Key/token sharing** — one `subscriptionId` presenting **>1 distinct
  `serverId`** within an overlapping active window (beyond a small migration
  grace).
- **Clone** — the same `serverId` seen on **multiple distinct `serverUrl`s** /
  request origins (copied DB/volume).
- **Seat overage / under-report** — `activeUserWorkspaceCount` or
  `distinctUserCount` materially exceeds the billed quantity, or seat reports drift
  inconsistently over time.
- **Dev-mode-in-prod** — `nodeEnv = development` on a licensed instance.
- **Modified build** — `imageDigest` not in the set of published, signed digests
  (low confidence).
- **Cadence / concurrency** — refresh frequency or **concurrent** refreshes for one
  subscription inconsistent with a single instance. This is the temporal twin of
  "sharing": derived purely from *when* calls arrive, so it holds even if every
  body field is spoofed, because each live copy must keep refreshing under short
  TTL.

## 6. Privacy & legal

- Enterprise-only and contractually disclosed in the enterprise terms.
- `adminContactEmail` is the only PII; covered by the DPA, purpose-limited to
  license administration/support, retention-limited, and secured. Never reused for
  other purposes.
- No free/community-user data and no CRM data are transmitted on this channel.
- The data sink is a security-sensitive asset; minimize fields and access
  accordingly (a breach of fleet telemetry would be worse than the abuse it
  guards against).

## 7. Out of scope for this note

- The dev-key / build-provenance hardening (e.g. a compiled `IS_OFFICIAL_RELEASE_BUILD`
  marker / secure-by-default dev-key trust) is **parked** and tracked separately —
  not part of this design.
- Image signing (cosign/Sigstore) is complementary (external verification &
  attribution) and specified elsewhere.
- Scoped/demo tokens for the `twenty-app-dev` image are a related but separate
  decision.

## 8. Implementation surface (when we build it)

- **twenty-server**
  - Gather counts + flags and extend the `/validate` and `/seats` request bodies
    (`enterprise-plan.service.ts`, `enterprise-key-validation.cron.job.ts`).
  - Add `serverId` to the calls; verify token instance-binding in `verifyJwt` and
    extend `EnterpriseValidityPayload`.
  - Possibly lower the default `ENTERPRISE_VALIDITY_TOKEN_DURATION_DAYS`.
- **twenty-website**
  - `signValidityToken`: include binding fields + (later) feature scope.
  - `/validate`, `/seats`: accept, persist, and correlate the new fields.
  - Detection store + rules (location TBD — possibly twenty-infra).

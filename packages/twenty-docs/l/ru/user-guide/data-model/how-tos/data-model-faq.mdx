## Управление объектами

<AccordionGroup>
  <Accordion title="Могу ли я удалить пользовательский объект?">
    Да, пользовательские объекты можно удалять. Вы также можете сначала деактивировать их, что скрывает объект и его данные из интерфейса, сохраняя при этом данные.
  </Accordion>

  <Accordion title="Могу ли я удалить стандартные объекты (Люди, Компании и т.д.)?">
    Нет, стандартные объекты не могут быть удалены. Их можно только деактивировать: это скрывает их из интерфейса, но сохраняет данные.
  </Accordion>

  <Accordion title="Сколько пользовательских объектов я могу создать?">
    Вы можете создать столько пользовательских объектов и полей, сколько вам нужно — цена не изменится.
  </Accordion>

  <Accordion title="Могу ли я переименовать стандартный объект?">
    Вы можете переименовать метку стандартных объектов (Люди, Компании, Сделки), но не их имена API. Имена API фиксированы для единообразия во всех рабочих пространствах Twenty.
  </Accordion>

  <Accordion title="Могу ли я изменить значок объекта?">
    Да, вы можете изменить значок как стандартных, так и пользовательских объектов в разделе **Настройки → Модель данных**.
  </Accordion>

  <Accordion title="Могу ли я переупорядочить объекты в левой панели навигации?">
    Пока нет. На данный момент последовательность объектов фиксирована, но эта функция планируется в будущем выпуске.
  </Accordion>

  <Accordion title="Могу ли я скрыть объекты из левой панели навигации?">
    Все активные объекты отображаются в навигации. Вы можете деактивировать ненужные объекты через **Настройки → Модель данных**.
  </Accordion>
</AccordionGroup>

## Возможности полей

<AccordionGroup>
  <Accordion title="Могу ли я изменить тип поля после создания?">
    Нет, типы полей нельзя изменить после создания. Если вам нужен другой тип, создайте новое поле с правильным типом, перенесите данные, затем деактивируйте старое поле.
  </Accordion>

  <Accordion title="Почему мне нужны разные формы для единственного и множественного числа?">
    Наш GraphQL API использует обе формы для различных операций:

    * `createPerson` (единственное число) для действий с одной записью
    * `createPeople` (множественное число) для массовых операций

    Это создает ограничения, когда формы единственного и множественного числа совпадают, но улучшает опыт разработчика.
  </Accordion>

  <Accordion title="Почему некоторые названия полей защищены?">
    Определенные названия полей, такие как `Тип` или `Приложение`, зарезервированы для системного использования. Выбирайте альтернативные названия, такие как `Категория` или `Классификация`.
  </Accordion>

  <Accordion title="Что происходит, когда я деактивирую поле?">
    * Поле скрывается из интерфейса
    * Существующие данные сохраняются
    * Вы по-прежнему можете получить доступ к полю через API
    * Существующие отношения сохраняются, но вы не можете создавать новые
    * Вы можете повторно активировать поле позже
  </Accordion>

  <Accordion title="Могу ли я сделать поле обязательным?">
    На данный момент сделать пользовательские поля обязательными нельзя. Все поля допускают пустые значения. Вы можете использовать рабочие процессы, чтобы обеспечить обязательность полей, отправляя уведомления или блокируя действия, когда поля пусты.
  </Accordion>

  <Accordion title="В чём разница между уникальным и обязательным?">
    * **Уникальное**: две записи не могут иметь одно и то же значение в этом поле
    * **Обязательное**: в поле должно быть значение (в настоящее время не поддерживается для пользовательских полей)
  </Accordion>

  <Accordion title="Могу ли я создавать поля-формулы?">
    Поля-формулы появятся в **Q1 2026**. До тех пор вы можете использовать рабочие процессы для автоматического вычисления и обновления значений полей.
  </Accordion>

  <Accordion title="Могу ли я использовать вложенные поля в моих объектах?">
    Вложенные поля появятся в **Q1 2026**. На данный момент вы можете использовать рабочие процессы для извлечения значений полей из связанных объектов. Например, чтобы отображать отрасль компании на записи Персоны, создайте пользовательское поле в Людях и используйте рабочий процесс для синхронизации значения.
  </Accordion>

  <Accordion title="Могу ли я переупорядочивать поля в объектах?">
    Переупорядочивание полей будет доступно вместе с настраиваемыми макетами в **Q4 2025**. На данный момент поля отображаются в алфавитном порядке.
  </Accordion>
</AccordionGroup>

## Связи

<AccordionGroup>
  <Accordion title="Могу ли я создать отношение, указывающее на тот же объект?">
    Да! Отношения, ссылающиеся на тот же объект, поддерживаются и рекомендуются для таких случаев, как иерархии учётных записей. Например, создайте отношение из Компаний к Компаниям, чтобы отслеживать родительские и дочерние учётные записи.
  </Accordion>

  <Accordion title="Могу ли я создавать отношения многие-ко-многим?">
    Отношения многие-ко-многим появятся в **H1 2026**. На данный момент создайте промежуточный объект с двумя отношениями один-ко-многим в качестве временного решения.

    Например, чтобы связать Людей и Проекты (многие-ко-многим), создайте объект "Назначения по проектам" с:

    * Связь с Людьми (много назначений → один человек)
    * Связь с Проектами (много назначений → один проект)
  </Accordion>

  <Accordion title="Что такое отношения к нескольким типам объектов (Morph-отношения)?">
    Они позволяют одному объекту связываться с несколькими различными типами объектов через одно поле. Например, Заметки могут одновременно быть прикреплены к Людям И Компаниям И Сделкам.

    Каждая Заметка одновременно связана с одним Человеком, одной Компанией и одной Сделкой.

    Подробнее см. в разделе [Поля связей](/l/ru/user-guide/data-model/capabilities/relation-fields).
  </Accordion>

  <Accordion title="Могу ли я создать несколько отношений между одними и теми же объектами?">
    Да, вы можете создать несколько отношений между одними и теми же двумя объектами. Например, у Компании могут быть отношения "Основной контакт" и "Контакт для выставления счетов" к Людям.
  </Accordion>

  <Accordion title="Что происходит с отношениями, когда я удаляю запись?">
    Когда вы удаляете запись, связь удаляется из связанных записей. Связанные записи не удаляются.
  </Accordion>

  <Accordion title="Могу ли я создать циклические отношения?">
    Хотя это технически возможно, циклических отношений (A → B → C → A) следует избегать, поскольку они могут вызывать путаницу и потенциальные проблемы с производительностью.
  </Accordion>
</AccordionGroup>

## Доступ и разрешения

<AccordionGroup>
  <Accordion title="Где я могу увидеть и изменить мою модель данных?">
    Перейдите в раздел **Настройки → Модель данных**, чтобы просмотреть и редактировать все свои объекты и поля.
  </Accordion>

  <Accordion title="Почему я не вижу Модель данных в настройках?">
    Обратитесь к администратору вашего рабочего пространства. Доступ к модели данных обычно ограничен только администраторами.
  </Accordion>
</AccordionGroup>

## Управление данными

<AccordionGroup>
  <Accordion title="Есть ли ограничение на количество записей?">
    Жёстких ограничений на количество записей нет. Однако очень большие наборы данных могут повлиять на производительность в некоторых представлениях. Используйте фильтры и представления, чтобы эффективно работать с большими наборами данных.
  </Accordion>

  <Accordion title="Могу ли я импортировать данные в пользовательские объекты?">
    Да, вы можете импортировать данные CSV в любой объект, включая пользовательские объекты. Процесс импорта поддерживает сопоставление пользовательских полей. См. [как подготовить файлы CSV](/l/ru/user-guide/data-migration/how-tos/prepare-your-csv-files).
  </Accordion>

  <Accordion title="Могу ли я экспортировать конфигурацию модели данных?">
    На данный момент встроенного экспорта конфигурации модели данных нет. Свяжитесь со службой поддержки, если вам нужно перенести модель данных между рабочими пространствами.
  </Accordion>
</AccordionGroup>

## Debugging with Workflow Runs

Use the **Workflow Runs** interface to debug issues:

- Access via the **Runs** panel in the workflow editor
- 個別の実行をクリックして各ステップの入力/出力データを確認
- 実行ステータス、エラーメッセージ、およびステップ間のデータフローを確認

## 一般的な問題と解決策

### ワークフローがトリガーされない

**問題**: ワークフローが期待通りに実行されない。

**解決策**:

- ワークフローが**アクティブ**（ドラフトモードではない）であることを確認
- トリガー設定がデータ構造と一致していることを確認
- レコードトリガーの場合、正しいオブジェクトとフィールドが選択されていることを確認
- Webhookトリガーの場合、URLと期待されるデータ形式を確認
- スケジュールドトリガーの場合、タイミング設定を確認

### フォームがアクセスできない

**問題**: フォームアクションが自動化されたワークフローで見つけにくい。

<Warning>
フォームは現在、手動トリガー専用に設計されています。 他のトリガーを持つワークフロー（レコード作成、更新など）では、ユーザーが期待する体験とは異なり、フォームにはワークフロー実行インターフェースからのみアクセス可能です。 通知センターは、2026年に自動化されたワークフローでフォームを適切にサポートするためにリリースされます。
</Warning>

**回避策**: フォームが必要な場合は手動トリガーを使用するか、自動フロー内でフォームを避けるようにワークフローを再構成します。

### 高クレジット消費

**問題**: ワークフローが予想以上のクレジットを消費します。

**一般的な原因と解決策**:

- **非効率なAPIコール**: 可能な場合は個別のリクエストではなくバッチAPIコールを行う
- **間違った手動トリガー設定**: 1回のワークフロー実行で複数のレコードを処理するには`Bulk`利用を使用します
- **フィルタの欠如**: 条件が満たされない場合にワークフローを停止するための条件付きロジックを追加
- **不要なステップ**: 冗長なアクションを削除しワークフロー論理を最適化する
- **リアルタイム対スケジュール**: 緊急でないプロセスにはスケジュールされたワークフローを使用する

### 同時ワークフロー制限

**問題**: ワークスペースごとに100の同時ワークフロー制限に達する。

<Warning>
You cannot run more than 100 workflows in parallel at any given time per workspace.
</Warning>

**解決策**:

- 手動トリガーのために`Bulk`の利用を使用して1回の実行で複数のレコードを処理
- スケジュールドトリガーを使用してワークフロー実行間に遅延を実装
- ワークフローを最適化し、実行時間を減らして同時実行を短縮
- 閑散期にバッチ処理を検討する

### APIレートリミット

**問題**: 外部API呼び出しがレートリミットにより失敗する。

**解決策**:

- 可能な限りリアルタイムトリガーではなくスケジュールドワークフローを使用
- コードアクションでのAPIコール間に遅延を実装
- 外部サービスがサポートしている場合はAPIリクエストをバッチ処理
- レートリミットエラーを監視し、タイミングを調整

### イテレータの問題

**問題**: イテレータアクションが期待通りに動作しない。

**解決策**:

- **注**: イテレータは現在ベータ版です。 設定→リリース→ラボの下でアクティベート
- 入力が前のステップからのレコードの配列であることを確認
- イテレーター内のアクションが正しく設定されていることを確認
- 最適なパフォーマンスのために`Bulk`手動トリガーとイテレーターを使用する

### 権限の欠如

**問題**: ワークフローセクションにアクセスできない。

<Warning>

ワークフローセクションが見えない場合、これは権限の問題です。 アクセスを許可するためにワークスペースの管理者に連絡してください。

## 最適化のヒント

### パフォーマンスのベストプラクティス

- **シンプルに始めて**徐々に複雑さを加える
- **手動トリガーに`Bulk`の利用**を使用して複数のレコードを効率的に処理
- **早めのフィルタ追加**で基準が満たされていない場合にワークフローを停止
- **APIコールをバッチ処理**し、緊急を要さないプロセスにはスケジュールドワークフローを使用
- **すべてのワークフローを表示するページからワークフロー実行を非表示**にして読み込み性能を改善
- **クレジット使用量を定期的に監視**し、設定→請求で確認

### エラープレベンション

- **ドラフトモードでテスト**してからワークフローを有効化
- **API応答を検証**し、フォールバックアクションを実装
- **メンテナンスを容易にするための詳細なステップ名**を使用
- **チームメンバー向けの複雑なロジックを文書化**する

## Workflow Triggers

Робочі процеси завжди починаються із одного тригера, який визначає, коли слід запускати автоматизацію.

### Запис створено

Starts the workflow when a new record is created in a selected object (People, Companies, Opportunities, or any custom object).

**Конфігурація**: Оберіть тип об'єкта для моніторингу нових записів.

### Запис змінено

Запуск робочого процесу відбувається при внесенні змін до існуючого запису.

**Конфігурація**:

- Оберіть тип об'єкта
- За необхідності вкажіть, які поля слід відстежувати для змін

### Запис змінено або створено

Starts the workflow when a record is either created or updated in a selected object.

**Чому це важливо**: Цей тригер особливо корисний, оскільки записи, створені різними методами, поводяться по-різному:

- **API/CSV imports**: Records are created with all fields populated immediately
- **Manual creation**: Records are created first, then fields are added in subsequent updates

**Конфігурація**:

- Оберіть тип об'єкта для моніторингу
- За необхідності вкажіть, які поля слід відстежувати для змін
- The workflow will trigger both on initial creation and any subsequent updates

### Запис видалено

Запуск робочого процесу відбувається при видаленні запису із об'єкта.

**Конфігурація**: Оберіть тип об'єкта для моніторингу видалень.

### Запустити вручну

Запуск робочого процесу відбувається за ініціативи користувача. Цей тригер можна викликати через меню Cmd+K або за допомогою спеціальної кнопки у верхньому навбарі.

**Конфігурація доступності**:
Обрати, як буде обрано запис:

- **Global**: No record is required to trigger this workflow. Робочий процес запускається з будь-якого місця (з будь-якого об’єкта) і не використовує запис(и) як вхідні дані.

- **Single**: The selected record(s) will be passed to your workflow. Це налаштовується для заданого об'єкта. Кілька записів можна вибрати перед запуском робочого процесу. Робочий процес виконуватиметься стільки разів, скільки обрано записів.

  <Warning>
  You cannot run more than 100 workflows in parallel at any given time.
  </Warning>

- **Масовий**: Вибраний запис(и) буде передано у ваш робочий процес. Це налаштовано для даного об’єкта. Кілька записів можна вибрати перед запуском робочого процесу. The workflow will run once, providing the entire list of records as input. This means the workflow needs to contain an Iterator action. This is best for people who want to optimize/limit the number of workflow runs.

**Додаткова конфігурація**:

- Select the target object (for Single and Bulk availability)
- Виберіть піктограму команди для тригера робочого процесу
- Configure navbar placement (Pinned or Not Pinned)

**Методи доступу**:

- Cmd+K menu to find and launch manual workflows
- Користувацька кнопка у верхній панелі навігації (якщо налаштовано)

### За розкладом

Запускає робочий процес на регулярній основі, яку ви визначаєте.

**Конфігурація**:

- Виберіть одиницю часу (хвилини, години, дні)
- Введіть значення або використовуйте спеціальні cron-вирази для розширеного планування

### Вебхук

Запускає робочий процес, коли отримано HTTP-запит (GET або POST) від зовнішньої служби.

**Конфігурація**:

- Отримайте унікальну URL-адресу webhook
- For POST requests, define the expected body structure
- Якщо потрібно, налаштуйте автентифікацію

## Дії в робочому процесі

Дії визначають, що відбувається після спрацювання тригера. Ви можете об’єднати кілька дій разом.

### Створити запис

Додає новий запис до вибраного об’єкта.

**Конфігурація**:

- Виберіть цільовий об’єкт
- Заповніть обов’язкові та необов’язкові поля
- Використовуйте дані з попередніх кроків для заповнення полів

**Вихідні дані**: Дані щойно створеного запису доступні для використання у наступних кроках.

### Оновлення запису

Змінює наявний запис у вибраному об’єкті.

**Конфігурація**:

- Виберіть цільовий об’єкт
- Виберіть конкретний запис для оновлення
- Виберіть поля для зміни та введіть нові значення

**Вихідні дані**: Оновлені дані запису доступні для використання у наступних кроках.

### Видалити запис

Видаляє запис з вибраного об’єкта.

**Конфігурація**:

- Виберіть цільовий об’єкт
- Виберіть конкретний запис для видалення

**Вихідні дані**: Дані видаленого запису все ще можуть бути використані у наступних кроках.

### Пошук записів

Знаходить записи в межах вибраного об’єкта, використовуючи критерії фільтра.

**Конфігурація**:

- Виберіть об’єкт для пошуку
- Встановіть критерії фільтра для звуження результатів
- Налаштуйте сортування та обмеження

**Вихідні дані**: Повертає запис, що відповідає умовам фільтра, для використання у наступних кроках.

**Найкраща практика**: Використовуйте гілки після пошуку записів для обробки сценаріїв «знайдено» та «не знайдено».

### Ітератор

Перебирає масив записів, отриманих з попереднього кроку, дозволяючи вам виконувати дії з кожним записом індивідуально.

**Конфігурація**:

- Виберіть масив записів з попереднього кроку (наприклад, результати пошуку записів)
- Визначте дії, які потрібно виконати з кожним записом в циклі
- Налаштуйте ім’я змінної для посилання на кожен запис в ітерації

**Приклад**: Пошук записів повертає 5 осіб, потім використовуйте Ітератор для відправки електронної пошти кожній особі або оновлення кожного запису індивідуально.

**Примітка**: Ітератор поки що в бета-версії. Активуйте його в Налаштуваннях → Релізи → Лабораторія.

### Фільтр

Фільтрує масив записів на основі вказаних умов, дозволяючи пройти лише записам, які відповідають критеріям.

**Конфігурація**:

- Виберіть масив записів для фільтрації
- Визначте умови та критерії фільтра
- Налаштуйте, які записи повинні проходити до наступних кроків

**Вихідні дані**: Повертає лише записи, які відповідають зазначеним умовам фільтра.

### Надіслати електронний лист

Надсилає електронну пошту з вашого робочого процесу.

**Передумови**: Додайте обліковий запис електронної пошти в Налаштуваннях > Облікові записи

**Конфігурація**:

- Введіть адресу електронної пошти одержувача
- Встановіть тему
- Напишіть текст повідомлення
- Використовуйте змінні з попередніх кроків для персоналізації

**Примітка**: Вкладення до електронної пошти будуть доступні у першому кварталі 2026 року.

### Код

Виконує користувацький JavaScript-код у вашому робочому процесі.

**Конфігурація**:

- Напишіть JavaScript-код в редакторі
- Отримуйте змінні з попередніх кроків
- Поверніть змінні для використання у наступних кроках
- Перевіряйте код безпосередньо в кроці

**Доступ**: Управління API-ключами в Налаштуваннях → API та Webhooks

### Форма

Під час виконання робочого процесу показується форма для збору введених користувачем даних.

**Конфігурація**:

- Визначте поля введення з типами, мітками та заповнювачами
- Налаштуйте правила валідації
- Встановіть заголовок та опис форми

**Вихідні дані**: Відповіді форми доступні для використання у наступних кроках.

<Warning>
Форми в даний момент розроблені лише для ручних тригерів. Для робочих процесів з іншими тригерами (запис створено, оновлено тощо), форми доступні лише через інтерфейс запуску робочого процесу, що не є очікуваним досвідом користувача. Центр повідомлень буде випущено у 2026 році для належної підтримки форм в автоматизованих робочих процесах.
</Warning>

### HTTP-запит

Надсилає запит до зовнішнього API в рамках вашого робочого процесу.

**Конфігурація**:

- Введіть URL-адресу кінцевої точки API
- Виберіть метод HTTP (GET, POST, PUT, PATCH, DELETE)
- Додайте необхідні заголовки та значення
- Додайте тіло запиту для запитів POST/PUT/PATCH
- Надайте зразок відповіді для попереднього перегляду структури

## Управління робочими процесами

### Створення робочих процесів

1. Натисніть «+ Додати робочий процес», щоб почати
2. Натисніть «Без назви», щоб назвати ваш робочий процес
3. Виберіть і налаштуйте тригер вашого робочого процесу
4. Додайте дії до вашого робочого процесу
5. Тестуйте та ітерація
6. Активуйте ваш робочий процес (зараз у чернетці), коли ви закінчите редагування

**Примітка**: Якщо ви не бачите розділ робочих процесів, це питання прав доступу. Зверніться до адміністратора вашої робочої області для надання доступу до робочих процесів.

### Статуси робочих процесів

- **Чернетка**: Редагується, ще не опублікована
- **Активний**: Жива версія, реагуюча на тригери
- **Деактивований**: Раніше активний, але зупинений вручну
- **Архівований**: Минули версії зберігаються в історії

### Активація робочих процесів

Натисніть **Активувати**, щоб опублікувати чернетку як нову версію. Це робить робочий процес відповідним для виконання при запуску тригером, але не виконує його негайно.

### Тестування робочих процесів

Перевірте робочі процеси, перш ніж активувати їх:

- Ручні тригери (коли не обрано запис)
- Індивідуальне тестування дій (особливо дій з кодом)
- Тестування в режимі чернетки, яке не активує робочий процес

### Виконання робочих процесів

**Запуск** - це запис виконання робочого процесу, який містить:

- Статус (успіх, невдача, виконання)
- Вихідні дані з кожного кроку
- Автор та часові мітки
- Повідомлення про помилки, якщо є

**Перегляд виконань**:

- Перевірте панель **Виконання** в редакторі робочого процесу
- Відкрийте перегляд **Виконання робочих процесів** для моніторингу всіх робочих процесів

**Порада з продуктивності**: Сховайте виконавчі записи з сторінки "Всі робочі процеси" та інших сторінок робочих процесів, щоб покращити продуктивність завантаження, оскільки велика кількість записів може уповільнити завантаження сторінки.

### Історія версій

- Перегляньте всі версії в полі **Версії**
- Натисніть на будь-яку версію для перегляду деталей
- Використовуйте **Використовувати як чернетку**, щоб відновити попередні версії
- Вирішіть конфлікти в чернетках за допомогою опцій перезапису або повернення

## Найкращі практики

### Організація робочих процесів

- **Описові назви**: Використовуйте чіткі, специфічні назви робочих процесів
- **Назви кроків**: Перейменовуйте кроки для опису їхньої функції
- **Документація**: Додавайте коментарі в дії з кодом
- **Категоризація**: Групуйте пов'язані робочі процеси логічно
- **Користувацькі поля**: Додайте поля до об’єкта Робочого процесу у вашій моделі даних (схожі на інші об’єкти), щоб організувати та категоризувати робочі процеси за допомогою користувацьких властивостей

### Оптимізація продуктивності

- **Мінімізація API-викликів**: Виконуйте пакетні операції, коли це можливо
- **Ефективні пошуки**: Використовуйте специфічні критерії фільтра
- **Обробка помилок**: Плануйте сценарії збоїв
- **Лімітування швидкості**: Дотримуйтеся лімітів зовнішніх API

### Управління потоком даних

- **Логіка гілок**: Використовуйте гілки після Пошуку записів
- **Використання змінних**: Використовуйте дані з попередніх кроків
- **Перевірка даних**: Перевіряйте введені дані в діях з кодом
- **Мапінг полів**: Плануйте трансформації даних ретельно

### Моніторинг та обслуговування

- **Регулярний моніторинг**: Перевіряйте виконання робочих процесів на помилки
- **Огляд продуктивності**: Аналізуйте час виконання і показники успішності
- **Управління оновленнями**: Тестуйте зміни в чернетці перед активацією
- **Командна координація**: Документуйте робочі процеси для членів команди

Для практичних прикладів використання цих функцій дивіться наші посібники по [Внутрішнім автоматизаціям](/l/uk/user-guide/workflows/internal-automations) та [Інтеграції із зовнішніми інструментами](/l/uk/user-guide/workflows/external-tool-integration).

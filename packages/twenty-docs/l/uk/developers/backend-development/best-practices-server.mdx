This document outlines the best practices you should follow when working on the backend.

## Дотримуйтесь модульного підходу

Бекенд дотримується модульного підходу, що є основним принципом при роботі з NestJS. Переконайтеся, що ви розбиваєте свій код на повторно використовувані модулі, щоб зберегти чисту та організовану кодову базу.
Кожен модуль повинен інкапсулювати конкретну функціональність або особливість і мати чітко визначену область. Цей модульний підхід дозволяє чітко розділяти задачі та видаляти непотрібну складність.

## Виставляйте сервіси для використання у модулях

Завжди створюйте сервіси з чіткою і єдиною відповідальністю, що покращує читаємість та підтримуваність коду. Name the services descriptively and consistently.

Ви також повинні виставити сервіси, які хочете використовувати в інших модулях. Виставлення сервісів для інших модулів можливе за допомогою потужної системи впровадження залежностей NestJS і сприяє слабкому зв'язуванню між компонентами.

## Уникайте використання типу `any`

Коли ви оголошуєте змінну як `any`, перевіряючий типу TypeScript не виконує перевірку типів, що дозволяє призначити змінній значення будь-якого типу. TypeScript використовує висновок типів, щоб визначити тип змінної на основі значення. При оголошенні як `any`, TypeScript більше не може вивести тип. Це ускладнює виявлення помилок, пов'язаних з типами, під час розробки, призводить до помилок у виконанні та робить код менш підтримуваним, менш надійним і важчим для розуміння іншими.

Ось чому все повинно мати тип. Тому, якщо ви створюєте новий об'єкт з ім'ям та прізвищем, ви повинні створити інтерфейс або тип, який містить ім'я та прізвище, що визначає форму об'єкта, яким ви оперуєте.

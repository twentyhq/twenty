## Debugging with Workflow Runs

Use the **Workflow Runs** interface to debug issues:

- Access via the **Runs** panel in the workflow editor
- 點擊個別運行以查看每個步驟的輸入/輸出數據
- 檢查執行狀態、錯誤信息和步驟之間的數據流

## 常見問題與解決方案

### 工作流程未觸發

**問題**：工作流程並未在預期時執行。

**解決方案**：

- 驗證工作流程是否 **激活**（而非草稿模式）
- 確認觸發配置是否與您的數據結構匹配
- 對於記錄觸發器，請確保選擇了正確的對象和欄位
- For Webhook triggers, verify the URL and expected data format
- 對於定時觸發器，請檢查定時設置

### 表單不可訪問

**問題**：表單動作難以在自動化工作流程中找到或訪問。

<Warning>
表單目前僅設計用於手動觸發。 對於具有其他觸發器的工作流程（記錄創建、更新等），表單僅可通過工作流程運行介面訪問，這不是預期的用戶體驗。 通知中心將於 2026 年發布，以正確支持自動化工作流程中的表單。
</Warning>

**解決方法**：當需要表單時使用手動觸發，或者重構工作流程以避免在自動流中使用表單。

### 高額信用消耗

**問題**：工作流程消耗的信用比預期多。

**常見原因和解決方案**：

- **不高效的 API 調用**：盡可能批量 API 調用，而不是單個請求
- **錯誤的手動觸發配置**：使用 `批量` 可用性而不是 `單一` 處理多個記錄於一次工作流程運行
- **缺少的篩選條件**：添加條件邏輯以在不符合條件時停止工作流程
- **不必要的步驟**：刪除冗余操作並優化工作流程邏輯
- **即時與排程**：對於非緊急流程使用排程工作流程

### 並發工作流限制

**問題**：達到每個工作區 100 個並發工作流的限制。

<Warning>
You cannot run more than 100 workflows in parallel at any given time per workspace.
</Warning>

**解決方案**：

- 對於手動觸發使用 `批量` 可用性，以便在一次運行中處理多個記錄
- 使用定時觸發在工作流程執行之間實施延遲
- 優化工作流以更快運行，並減少並發執行時間
- 考慮在非高峰時間批量處理

### API 速率限制

**問題**：外部 API 調用由於速率限制失敗。

**解決方案**：

- 在可能的情況下使用排程工作流代替即時觸發器
- 在代碼操作中實施 API 調用之間的延遲
- 當外部服務支援批量時批量 API 請求
- 監控工作流程執行，以檢查速率限制錯誤並調整時間

### 迭代器問題

**問題**：迭代器操作未按預期工作。

**解決方案**：

- **注意**：迭代器目前處於測試階段。 在 設置 → 版本 → 實驗室下激活
- 確認輸入是來自上一步的記錄數組
- 檢查迭代器中的操作是否已正確配置
- 將迭代器與 `批量` 手動觸發一起使用以獲得最佳性能

### 缺少權限

**問題**：無法訪問工作流程部分。

<Warning>

如果您看不到工作流程部分，這是由於權限問題。 聯繫您的工作區管理員以授予您訪問工作流程的權限。

## 優化提示

### 性能最佳實踐

- **從簡單開始**，逐漸增加複雜性
- **使用 `批量` 可用性** 手動觸發，以高效處理多個記錄
- **Add filters early** to stop workflows when criteria aren't met
- **批量 API 調用** 並使用排程工作流程以處理非緊急流程
- **Hide workflow runs** from pages showing all workflows to improve loading performance
- **定期監控信用使用情況** 在設置 → 計費

### 錯誤預防

- **在激活工作流程之前在草稿模式下測試**
- **驗證 API 響應** 並執行後備操作
- **使用描述性的步驟名稱** 以便於維護
- **為團隊成員記錄複雜的邏輯**

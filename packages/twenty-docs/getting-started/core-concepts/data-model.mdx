The **data model** is how you describe your business to Twenty. Objects are the kinds of things you track. Fields are what you record about each one. Relations connect them.

Most CRM disappointment traces back to this layer. A system that models the business wrong stays wrong, because everyone is too busy working around it to fix it. The test of a good model is that questions come out as filters rather than projects. Which orders still reference the old artwork. Which deals over fifty thousand close this quarter. Those are one click each when the model matches the business.

## Objects

An **object** is one kind of thing you track, one tab in spreadsheet terms. An entry inside an object is a **record**, one specific company, one specific deal. Twenty ships with the objects every business needs.

- **Companies** and **People**, who they are and where they work
- **Opportunities**, Twenty's term for deals
- **Tasks** and **Notes**, the work and context around the rest

The set that ships is deliberately small, because the interesting objects are yours. A packaging supplier lives off print runs. Each run has a design, a quantity, a ship date and a destination plant, so the supplier creates a Print Runs **custom object**. The result is first-class. Its own place in the sidebar, its own pages and views, its own API endpoints and its own permissions. Custom data in most systems is an afterthought, bolted onto someone else's schema. In Twenty it's the point.

## Fields

A **field** is one piece of information on a record, one column in spreadsheet terms. An order quantity, a plant address, the current artwork version. Every field has a **type**, a commitment that buys behavior.

- **Currency** fields sum, so a dashboard totals open deals without anyone maintaining the number.
- **Select** fields hold one value from a fixed list, which is exactly what pipeline columns are built from.
- **Date** fields put records on a calendar.
- **Files** fields hold documents, so the current artwork PDF lives on the record it describes.

There are more than twenty types. A few fields Twenty maintains itself on every record, including who created it and when it last changed.

## Relations

A **relation** is a field that holds another record rather than a plain value. On a purchasing manager's record, the Company field holds the Harbor Foods record itself, not the text `Harbor Foods`. That distinction is what keeps every fact in one place. The current artwork lives on the Harbor Foods record. Every print run points at the company, so every print run reads the artwork from the one place it exists. A rebrand becomes a one-field edit. Nothing downstream ships stale.

Relations run in both directions. Open Harbor Foods and every person, deal, note and print run connected to it is on the page. Open any of those and the company is one click back.

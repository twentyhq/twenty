The **data model** is how you describe your business to Twenty. Underneath sits a real relational database. Each workspace is a PostgreSQL schema where objects are tables, fields are typed columns and relations are foreign keys. Twenty generates the rest from the model you define: the pages, the views, the API, the permissions.

## Objects

An **object** is a table and a **record** is a row in it. Twenty ships with the standard set.

- **Companies** and **People**, the accounts and their contacts
- **Opportunities**, Twenty's term for deals
- **Tasks** and **Notes**, the work and context around the rest

The set is deliberately small because the interesting objects are yours. A **custom object** is a table you define yourself, and it gets the same machinery as Companies: its own place in the sidebar, its own pages and views, its own API endpoints, its own permissions and its own workflow triggers. A custom object can hold whatever your business runs on: support tickets, products, contracts, properties.

## Fields

A **field** is a column with a type. The type is a commitment that buys behavior.

- **Currency** fields sum, so a dashboard totals open deals without anyone maintaining the number.
- **Select** fields hold one value from a fixed list, which is what pipeline columns are built from.
- **Date** fields put records on a calendar.
- **Files** fields hold documents, so a contract PDF lives on the record it belongs to.

There are more than twenty types in total.

| Kind | Types |
|---|---|
| Plain | Text, Rich Text, Number, True/False, Date, Date and Time, Rating |
| Structured | Currency, Select, Multi-select, Emails, Phones, Links, Address, Full Name |
| Technical | Relation, Files, JSON, Array, Unique ID |

Twenty also maintains a few system columns on every record, including creation and update timestamps and who created it.

## Relations

A **relation** is a foreign key with an interface. The Company field on a person's record holds a reference to the company row, not the company's name as text. That reference keeps every fact in one place. Rename a company and every person, opportunity and note that references it already shows the new name, because none of them ever held a copy.

Relations resolve in both directions. Open a company and every person, opportunity, task and note that references it is on the page. Open any of those and the company is one click back.

The standard objects arrive already connected. A company has many people, an opportunity belongs to a company. Custom objects join the same graph, including many-to-many relations where both sides hold lists of each other.

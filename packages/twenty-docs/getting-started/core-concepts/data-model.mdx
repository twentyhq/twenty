The **data model** is how you describe your business to Twenty. Underneath sits a real relational database. Each workspace is a PostgreSQL schema where objects are tables, fields are typed columns and relations are foreign keys. Twenty generates the rest from the model you define: the pages, the views, the API, the permissions.

## Objects

An **object** is a table and a **record** is a row in it. A packaging supplier that sells to the food brand Harbor Foods keeps each brand as a record in Companies and each buyer there as a record in People. Twenty ships with the standard set.

- **Companies** and **People**, the accounts and their contacts
- **Opportunities**, Twenty's term for deals
- **Tasks** and **Notes**, the work and context around the rest

The set is deliberately small because the interesting objects are yours. The supplier lives off print runs, so it creates a Print Runs **custom object**. Behind the scenes that's a real table in the workspace schema, with the same machinery as Companies: its own place in the sidebar, its own pages and views, its own API endpoints, its own permissions and its own workflow triggers. A custom object can hold whatever your business runs on. Projects, support tickets, products, contracts, properties.

## Fields

A **field** is a column with a type. The type is a commitment that buys behavior.

- **Currency** fields sum, so a dashboard totals open deals without anyone maintaining the number.
- **Select** fields hold one value from a fixed list, which is what pipeline columns are built from.
- **Date** fields put records on a calendar.
- **Files** fields hold documents, so Harbor Foods' current artwork PDF lives on its record.

There are more than twenty types in total.

| Kind | Types |
|---|---|
| Plain | Text, Rich Text, Number, True/False, Date, Date and Time, Rating |
| Structured | Currency, Select, Multi-select, Emails, Phones, Links, Address, Full Name |
| Technical | Relation, Files, JSON, Array, Unique ID |

Twenty also maintains a few system columns on every record, including creation and update timestamps and who created it.

## Relations

A **relation** is a foreign key with an interface. The Company field on a purchasing manager's record holds a reference to the Harbor Foods row, not the text `Harbor Foods`. That reference keeps every fact in one place. The current artwork lives on the Harbor Foods record. Every print run references the company, so each one reads the artwork from the one place it exists. A rebrand becomes a one-field edit. Nothing downstream ships stale.

Relations resolve in both directions. Open Harbor Foods and every person, deal, note and print run that references it is on the page. Open any of those and the company is one click back.

The standard objects arrive already connected. A company has many people, an opportunity belongs to a company. Your custom objects join the same graph, including many-to-many relations where both sides hold lists of each other.
